## 1. 사전 준비

AI 기반 코딩에서 일관된 결과를 얻으려면 첫 번째 코드 줄을 작성하기 전부터 준비가 필요합니다.

AI 개발 워크플로우를 구축하면서 느끼는 점은, AI는 작업 환경을 설계해 줄 때 더 나은 코드를 작성한다는 것입니다. 이러한 방식을 **하네스 엔지니어링** 이라고 부르는 경우도 있습니다. 핵심은 자유로운 형식의 티켓이 아닌 구조화된 컨텍스트를 제공하는 데 있습니다.
<br>
<br>

## 2. 문제점, 해결책, 그리고 효과

### 2.1 문제점: 입력이 모호하면 출력도 모호

AI를 활용하여 소프트웨어 개발 수명주기(SDLC)를 가속화하기 위해, Jira의 기능 설명을 AI 코딩 도구에 붙여넣고 AI 에이전트에게 "구현해줘"라고 명령하고 결과를 기다리는 경우입니다.

* 결과는 예측할 수 없음
  + 인공지능이 성공하는 경우도 있었지만, 파일 경로를 잘못 해석하거나, 존재하지 않는 API를 만들어내거나, 완전히 엉뚱한 모듈을 수정하는 경우도 있음
  + 실패 원인은 비슷하며, 인공지능이 코드베이스를 제대로 살펴보지 않고 추측만 했던 경우임

* 근본적인 문제는 AI의 코딩 능력이 아니라 사용자가 제공한 입력값
  + "SBOM용 CSV 내보내기 추가"와 같은 자유 형식의 Jira 티켓은 해결 가능성을 무궁무진하게 만듦
  + 어떤 서비스가 소프트웨어 명세서(SBOM)를 처리하는지, 기존 내보내기 패턴은 무엇인지, 테스트 파일은 어디에 저장되는지 등 여러 질문에 대한 답이 없기 때문에 AI는 최적의 해결책을 제시하지 못함
<br>

### 2.2 해결책 - 2단계 워크플로

문제를 계획과 실행이라는 두 단계로 나누고 각 단계에 고유한 제약 조건을 설정하여, 각 단계에서 해결 범위를 제한함으로써 결과물의 일관성을 높일 수 있습니다.

#### 2.2.1 1단계: 저장소 영향력 맵(지도) 생성

* 작업을 생성하기 전에 AI를 사용하여 실제 코드베이스를 스캔하고 저장소 영향력 맵(지도)을 생성
  + 이 맵은 저장소를 유일한 정보 소스로 간주하여 작성됨
  + AI는 언어 서버 프로토콜(LSP) 및 모델 컨텍스트 프로토콜(MCP) 서버를 사용하여 저장소 구조를 검사
  + 기호를 찾고, 참조를 추적하고, 패턴을 검색
  + 그런 다음 다음과 같은 맵을 생성
    ```yaml
    trustify (backend):
      changes:
        - Add CSV serialization for SBOM query results
        - Add GET /api/v2/sbom/export endpoint
    
    trustify-ui (frontend):
      changes:
        - Add "Export CSV" button to SBOM list toolbar
    ```
* 중요한 단계
  + 다른 모든 작업이 진행되기 전에 사람이 이 맵을 검토
  + AI가 잘못된 모듈을 선택했거나, 저장소를 놓쳤거나, 존재하지 않는 엔드포인트 패턴을 만들어낸 경우, 여기서 사용자가 이를 잡아내어 잘못된 작업과 부실한 코드를 방지
* 이 체크포인트는 흔히 발생하는 오류 유형을 조기에 포착
  + 이러한 오류는 구조적으로 잘못된 계획으로 인해 컴파일은 되지만 잘못된 문제를 해결하는 코드를 생성하는 경우

#### 2.2.2 2단계: 구조화된 작업 템플릿

* 영향도 맵(지도)이 승인되면 각 작업 단위는 엄격한 템플릿을 따르는 Jira 작업으로 전환됨
  ```jira
  ## Repository
  trustify
  
  ## Description
  Add a CSV export endpoint for SBOM query results.
  
  ## Files to Modify
  - `modules/sbom/src/service.rs`—add CSV serialization method
  - `modules/sbom/src/endpoints.rs`—add GET handler
  
  ## Implementation Notes
  Follow the existing JSON export pattern in `SbomService::export_json()`.
  Reuse the `QueryResult` type from `modules/sbom/src/model.rs`.
  
  ## Acceptance Criteria
  - [ ] GET /api/v2/sbom/export?format=csv returns valid CSV
  - [ ] Existing JSON export still works
  
  ## Test Requirements
  - [ ] Integration test in `modules/sbom/tests/` following existing test patterns
  ```
* 모든 필드는 의도적으로 설계됨
  + **Repository**
    - AI를 단일 저장소로 제한하여 저장소 간 혼동을 방지
  + **Description**
    - 작업이 무엇을 의미하는지 요약
  + **Files to Modify**
    - 목록에 추측이 아닌 분석 단계에서 발견된 실제 경로가 포함
  + **Implementation Notes**
    - 실제 기호 이름과 기존 패턴을 참조
    - AI가 *Follow the existing JSON export pattern in `SbomService::export_json()`*를 읽을 때, 해당 함수를 찾아보고 구조를 모방 가능
  + **Acceptance Criteria**
    - 인공지능에게 검증할 구체적인 체크리스트를 제공
  + **Test Requirements**
    - 이 문서에서는 해당 작업에 필요한 테스트 범위에 대한 지침을 제공
* 이 템플릿은 계획과 실행 간의 계약서와 같음
  + 작업을 실행하는 AI는 아키텍처 관련 결정을 내릴 필요가 없음
  + 그러한 제약 조건은 영향 지도에 이미 설정되어 있음
<br>

### 2.3 효과

**기본 원리**
* 해법의 범위를 좁힐수록 결과가 더 예측 가능해짐
  + AI가 실제 코드를 기반으로 계획을 수립하면 타당한 계획을 생성
  + 실제 파일 경로와 심볼 이름이 포함된 구조화된 사양을 기반으로 구현하면 목표에 맞는 코드 변경 사항을 생성
* 이 과정을 반복하며 피드백 루프를 강화
  + 초기 버전의 계획 수립 기능은 검토 단계 없이 방대한 분석 결과를 한꺼번에 쏟아냄
  + 그 결과, AI는 "제가 발견한 내용은 다음과 같습니다"라고 표시한 후 곧바로 작업을 생성하는 데로 넘어감
  + 영향 지도에 대한 사람의 검토를 위해 명시적으로 일시 정지하는 기능을 추가한 것은 작은 변화였지만, 오류를 눈에 띄게 줄여줌
* 마찬가지로 작업 템플릿도 진화
  + 첫 번째 버전에는 일반적인 메모 필드가 있었음
  + 이 필드를 구현 메모로 이름을 바꾸고 실제 코드 기호를 참조하도록 요구하면 결과가 달라림
  + 모호한 설명은 계획 단계가 실제 분석에 기반하지 않았음을 나타냄
<br>

## 3. 요약 및 다음 단계 

### 3.1 핵심 요약

AI 기반 개발 워크플로우를 구축하려는 경우
* 코드를 작성하는 것부터 시작하는 것이 아닌, 환경을 설계하는 하네스 엔지니어링부터 시작
  + AI가 계획을 세우기 전에 실제 코드를 살펴 확인
    - 심볼 분석을 기반으로 구축된 저장소 영향 맵은 모든 하위 작업에 실제 코드를 기반으로 함
  + AI가 구현할 수 있도록 구조화된 제약 조건을 제공
    - 파일 경로, 심볼 이름, 따라야 할 기존 패턴 등 입력이 구체적일수록 출력의 일관성이 높아짐
  + 계획과 구현 사이에 사람의 검토 과정을 거침
    - 세 줄짜리 영향도 맵에서 잘못된 가정을 잡아내는 데 드는 비용은 풀 리퀘스트에서 이를 잡아내는 데 드는 비용보다 훨씬 적음
  + Jira 기능의 품질에 투자
    - 명확한 범위, 구체적인 승인 기준, 관련 컨텍스트를 포함하는 잘 작성된 기능 설명은 AI가 생성하는 계획과 코드의 품질을 직접적으로 향상시킴
    - 구조화된 입력이 구조화된 출력으로 이어지는 원칙은 요구사항 자체에도 적용됨

* 즉각적인 엔지니어링에서 활용 엔지니어링으로의 전환은 개발 주기에서 AI를 다루는 방식의 변화를 의미
  + AI를 마법의 상자처럼 여기는 대신 구조화된 환경 내의 구성 요소로 취급하기 시작하면 결과는 예측 가능하고 재현 가능해짐
  + AI를 코드베이스에 기반을 두고 명확한 구현 제약 조건을 제공함으로써 변덕스러운 보조 도구를 신뢰할 수 있는 기여자로 바꿀 수 있음
  + 구조를 넣으면 구조가 나옴 (모호함을 넣으면, 모호함이 나옴)
<br>

### 3.2 다음 단계

* 해당 저장소를 규칙을 포함한 모든 정보의 단일 소스로 만듦
  + 스타일 가이드, 명명 규칙, 아키텍처 설계 결정 사항 등을 저장소 자체에 통합(예:  CLAUDE.md또는  CONVENTIONS.md)
  + AI가 코드베이스를 읽을 때 이러한 규칙을 자동으로 인식하므로 프롬프트에 복사 붙여넣기를 할 필요가 없음
* 하네스를 유지 관리해야 하는 소프트웨어처럼 다룸
  + 스킬, 프롬프트, MCP 구성은 코드임
  + 버전을 관리하고, PR에서 검토하고, 변경 사항이 생기면 리팩토링
  + 오래된 프롬프트는 오래된 테스트처럼 쓸모없어짐
* 에이전트와의 피드백 루프를 완성
  + 생성된 계획이나 PR이 잘못된 경우, 입력값에서 오류를 찾아냄
  + 예를 들어, 누락된 제약 조건, 모호한 승인 기준, AI가 인식하지 못한 기호 등을 확인 필요
  + 출력뿐만 아니라 시스템의 근본적인 문제까지 해결 필요
* MCP를 통해 에이전트의 도구 상자를 확장
  + 새로운 MCP 통합(예: 지속적 통합(CI) 상태, 배포 로그 및 런타임 메트릭)은 AI에 또 다른 실제 데이터 소스를 제공
