## 1. 추론 서비스 성능과 기준

### 1.1 추론 서비스 성능

기업들이 대규모 언어 모델(LLM)을 파일럿 단계에서 프로덕션 단계로 이전함에 따라, 일관된 추론 성능을 유지하는 것이 주요 운영 과제가 됩니다. vLLM은 기본적으로 효율적이지만, 단순 벤치마크 수준을 넘어서면 사용자는 종종 기대와 다른 성능 문제를 경험합니다.

vLLM은 서버의 내부 상태를 파악할 수 있는 포괄적인 메트릭 세트를 제공하여 성능 문제의 근본 원인을 진단하는 데 도움을 줍니다. 이 글에서는 vLLM 배포 환경의 성능을 개선하는 데 도움이 되는 진단 워크플로를 제공합니다.
<br>

### 1.2 성공의 기준을 정의

진단 지표를 검토하기 전에 다음 질문을 해봅니다. 성능 목표는 무엇일까?

이 질문에 대한 답을 통해 어떤 지표가 가장 중요하고 어떤 개선 조치를 우선시해야 하는지 결정할 수 있습니다.
최적화 전략은 일반적으로 다음 워크로드 프로필 중 하나에 따라 달라집니다.

* 처리량 중심(오프라인/배치)
  + 데이터 추출, 요약 또는 문서 분류와 같은 대량의 요청을 처리
  + 개별 요청 지연 시간보다 전체 작업 완료 시간이 더 중요
  + 초당 요청 처리량을 최적화 필요

* 지연 시간 중심(온라인/대화형)
  + 사용자가 스트리밍 응답을 기다림
  + 챗봇, 코드 자동 완성, 실시간 음성 응답 등
  + 첫 번째 토큰 수신 시간(TTFT)과 토큰 간 지연 시간(ITL)은 사용자 경험에 직접적인 영향을 미침
  + 예상 동시 접속자 수에서 낮은 지연 시간을 최적화 필요

* 버스트형(준온라인)
  + 부하가 예측 불가능하며 유휴 상태에서 급증 상태로 끊임없이 전환
  + 처리량이나 최소 지연 시간보다는 유연한 확장성과 빠른 콜드 스타트가 더 중요

배치 파이프라인을 실행하는 경우 응답 시간이 길더라도 전혀 문제가 되지 않을 수 있습니다. 반대로 챗봇을 운영하는 경우에는 동시 접속자 수가 적을 때는 요청 지연 시간이 괜찮아 보일 수 있지만, 서버가 포화 상태가 되면 허용할 수 없는 수준이 될 수 있습니다. 워크로드를 사전에 명확히 정의하면 이후 단계에서 올바른 지표에 집중할 수 있습니다.

성능 목표를 설정했다면, 지연이 발생하는 지점을 파악하여 문제 해결 프로세스를 시작할 수 있습니다.
<br>
<br>

## 2. 문제 해결 프로세스

### 2.1 증상 분리: TTFT 대 ITL 비교

#### 2.1.1 전체 종단 간(E2E) 지연 시간

일반적으로 성능을 파악하는 첫 번째 지표
* 서비스 수준 목표(SLO)를 정의하고 전반적인 사용자 경험을 이해하는 데 중요한 지표
* E2E 지연 시간을 통해 요청이 느린지 여부를 알 수 있지만, 느린 원인을 파악하려면 요청 처리의 두 가지 구성 요소에 대한 측정값, TTFT와 ITL을 자세히 살펴야 함
  + 첫 번째 토큰 생성 시간 (TTFT; Time to First Token)
    - 서버가 요청을 수신한 시점부터 첫 번째 토큰이 생성될 때까지의 시간
    - 여기에는 대기열 지연(서버 과부하로 인해 요청이 대기하는 시간)과 사전 입력 단계(엔진이 입력 프롬프트를 처리하는 단계)가 포함
    - 서버 과부하 상태에서는 TTFT가 대기열 지연 시간에 의해 좌우되는 경우가 많음
  + 토큰 간 지연 시간(ITL: Inter-Token Latency)
    - 디코딩 단계에서 각 토큰이 순차적으로 생성될 때까지의 시간 간격을 측정
    - ITL은 텍스트가 사용자에게 스트리밍될 때 얼마나 부드럽게 표시되는지를 나타냄
    - ITL이 높다는 것은 일반적으로 하드웨어 대역폭 제한이나 배치 처리 내 높은 동시성을 의미

사전 채우기(prefill)와 디코딩(decode)는 서로 상호 작용
* 포화 상태인 서버에서 긴 사전 채우기 (prefill) 작업이 실행되면 다른 진행 중인 요청의 디코딩(decode) 단계가 일시적으로 지연되어 배치에 있는 모든 요청의 ITL이 급증할 수 있음
* vLLM은 청크 사전 채우기(기본적으로 활성화됨)를 통해 이를 완화

청크 사전 채우기
* 긴 사전 채우기 작업을 더 작은 청크로 나누어 디코딩 단계와 번갈아가며 처리
* 새로운 긴 프롬프트 요청이 도착할 때 주기적인 ITL 급증이 발생하는 경우 청크 사전 채우기가 의도대로 작동하고 있는 것
* 그러나 이러한 급증은 `--max-num-batched-tokens` 설정을 조정해야 함을 나타낼 수 있음

#### 2.1.2 지연 시간 지표

다음 세 가지 지표를 살펴보면 병목 현상을 좁힐 수 있음
* E2E 요청 지연
  + vLLM에서 전체 요청이 처리되는 데 걸리는 총 시간
  + 프로메테우스 쿼리
    ```sql
    histogram_quantile(0.5, sum by(model_name, pod, le)(
        rate(vllm:e2e_request_latency_seconds_bucket{}[1m]))
    )
    ```

* TTFT
  + 대기열 시간 + 사전 채우기 단계 시간
  + 프로메티우스 쿼리
    ```sql
    histogram_quantile(0.5, sum by(model_name, pod, le)(
      rate(vllm:time_to_first_token_seconds_bucket{}[1m]))
    )
    ```
* ITL
  + 생성된 토큰 간 시간(디코딩 단계)
  + 프로메테우스 쿼리
    ```sql
    histogram_quantile(0.5, sum by(model_name, pod, le)(
      rate(vllm:inter_token_latency_seconds_bucket{}[1m]))
    )
    ```

예
* E2E 지연 시간과 TTFT는 높지만 ITL이 낮다면 서버에 과부하가 걸려 요청이 큐에 쌓이는 것일 수 있음
* 또한 입력 프롬프트가 너무 길어 사전 입력 시간이 길어지는 것도 원인일 수 있음

고려 사항
* 이러한 지표는 서버 측 지표이며 네트워크 또는 인그레스 지연 시간은 포함하지 않음
* 클라이언트에서 측정한 지연 시간이 vLLM에서 보고하는 값보다 현저히 높다면 네트워크 경로, 로드 밸런서 또는 프록시 계층에 문제가 있을 가능성이 높음
<br>

### 2.2 서버 포화 상태 감지

#### 2.2.1 서버 큐 깊이 및 배치 크기 확인

이전 단계에서 얻은 지연 시간 지표를 이해하기 위해 서버 큐 깊이와 실행 중인 배치 크기 확인 필요
* 높은 TTFT는 요청이 긴 큐에서 대기하는 경우 발생 가능
* ITL은 일반적으로 배치 크기에 비례

vLLM은 처리량을 극대화하기 위해 연속 배치 처리를 사용
* 실행 중인 요청 수가 증가함에 따라 각 사용자의 개별 지연 시간도 일반적으로 증가하는 필연적인 절충점이 존재
* 실행 중인 요청과 대기 중인 요청 간의 관계를 모니터링함으로써 서버가 포화 상태인지, 아니면 추론의 특정 단계가 단순히 연산 병목 현상인지 판단할 수 있음

#### 2.2.2 서버 상태 지표

**실행 중인 요청**과 **대기 중인 요청**의 균형을 모니터링하여 서버 용량 한계에 도달하는 시점을 파악
* 실행 중인 요청
  + GPU에서 활발하게 처리 중인 요청(배치 크기)
  + 프로메테우스 메트릭 `vllm:num_requests_running`을 사용
* 대기 중인 요청
  + 리소스 제한으로 인해 큐에 대기 중인 요청(큐 깊이)
  + 프로메테우스 메트릭 `vllm:num_requests_waiting`을 사용

예
* `num_requests_waiting`이 지속적으로 `0`보다 크면 요청이 큐에서 대기하는 시간이 길어져 TTFT가 높아짐
* `num_requests_waiting`이 `0`이지만 TTFT가 여전히 높으면 지연은 사전 채우기 단계 자체에 필요한 시간(긴 프롬프트 처리) 때문일 가능성이 높음
  + 이는 vLLM 서버가 컴퓨팅 병목 현상을 겪고 있음을 나타냄

#### 2.2.3 서버 상태 로그

vLLM은 실행 중이거나 대기 중인 요청 수, 키-값(KV) 캐시 메트릭을 포함한 엔진 상태의 스냅샷을 주기적으로 기록하고, 이러한 로그를 통해 서버 상태 확인
* 정상적인 서버(적절하게 사용되지만 과부하가 걸리지 않은 상태)
  ```log
  Engine 000: Running: 39 reqs, Waiting: 0 reqs, GPU KV cache usage: 68.9%, Prefix cache hit rate: 29.7%
  ```
  + 대기 중인 요청이 없고 KV 캐시 사용률이 90% 미만

* 포화 상태의 서버
  ```log
  Engine 000: Running: 60 reqs, Waiting: 21 reqs, GPU KV cache usage: 99.8%, Prefix cache hit rate: 32.2%
  ```
  + KV 캐시 용량이 거의 100%에 도달하여 요청이 "대기" 큐에 쌓임
  + 이러한 사용자는 프롬프트 크기에 관계없이 TTFT(Time To Fly Time)가 높아짐

> [!NOTE]
> "적절한" 배치 크기는 하드웨어 및 모델 선택에 따라 다릅니다. 대형 GPU에서 실행되는 소형 모델의 경우 vLLM은 허용 가능한 지연 시간으로 수백 개의 요청을 배치 처리할 수 있습니다.
<br>

### 2.3 VRAM 및 KV 캐시 상태 평가

#### 2.3.1 vLLM 및 GPU 메모리

**GPU 메모리**
* 시스템이 동시에 처리할 수 있는 요청 수를 제한하는 주요 요인
* 주로 정적 모델 가중치와 동적 KV 캐시를 저장하는 데 사용
* 또한, CUDA 그래프와 활성화 함수도 소량의 GPU 메모리를 사용

**GPU 메모리와 모델 가중치**
* 모델 가중치가 하드웨어 용량을 초과하면 요청을 동시에 처리할 수 있는 KV 캐시 공간이 부족해짐
* 이러한 "메모리 부족 현상"이 2단계에서 확인한 대기 중인 요청의 가장 일반적인 원인
* 양자화는 이러한 부담을 크게 줄일 수 있음

**vLLM이 지원하는 양자화 모델**
* 양자화된 가중치(예: FP8, AWQ 또는 GPTQ)를 사용하는 모델 실행
* `--kv-cache-dtype fp8` 옵션을 사용한 KV 캐시 양자화

> [!NOTE]
> 이러한 양자화 기법은 가중치와 토큰의 메모리 사용량을 줄여 엔진이 더 큰 배치 처리를 할 수 있도록 공간을 확보해줍니다.

#### 2.3.2 메모리 부족 현상의 심각도를 진단을 위한 주요 상태 지표

* KV 캐시 점유율
  + 현재 사용 중인 KV 캐시의 비율
  + 이 값이 지속적으로 100%에 가까우면 새 요청은 대기
  + 프로메테우스 메트릭 `vllm:kv_cache_usage_perc`를 사용

* 총 선점 횟수
  + 메모리 확보를 위해 생성 도중 중단된 요청의 누적 횟수
  + 이는 심각한 메모리 스래싱을 ​​나타내며 사용자에게 상당한 지연 시간 급증을 초래
  + 이 카운터가 증가하는 경우 스케줄러가 KV 캐시가 감당할 수 있는 것보다 더 많은 요청을 허용하지 않도록 --max-num-seqs 값을 줄임
  + 프로메테우스 메트릭 `vllm:num_preemptions_total`을 사용

#### 2.3.3 시작 시 용량 점검

* vLLM은 시작 시 모델 및 오버헤드를 로드한 후 KV 캐시에 남은 메모리 용량을 정확하게 계산
  + 이를 통해 트래픽이 발생하기 전에 모델 크기가 적절한지 판단

* 예: 메모리 제약이 있는 모델(H100 GPU 하나에서 실행되는 32B FP16 모델)
  ```
  Available KV cache memory: 4.46 GiB
  Maximum concurrency for 10,000 tokens per request: 1.83x
  ```
  + 가중치와 오버헤드가 80GB GPU 메모리의 거의 전부를 소모
  + 캐시에 남은 메모리가 4.4GiB에 불과하므로 서버는 동시에 두 명 이상의 사용자가 접속할 경우 제대로 작동하지 못할 수 있음

* 로그에 사용 가능한 KV 캐시 메모리가 매우 낮게 표시되는 경우
  + 양자화된 모델, 더 작은 모델 또는 더 많은(또는 더 큰) GPU를 사용하여 활성 요청을 위한 공간을 확보하는 것을 고려

#### 2.3.4 접두사(prefix) 캐싱

* 많은 사용 사례에서 프롬프트는 동일한 접두사로 시작
  + 예를 들어, 여러 차례 대화가 이어지는 채팅 환경에서는 대화 기록이 후속 메시지의 프롬프트에 포함
  + vLLM 접두사 캐싱은 이러한 세그먼트를 KV 캐시에 저장하여 재처리할 필요가 없도록 함
  + vLLM에서는 접두사 캐싱이 기본적으로 활성화
  + 높은 캐시 적중률은 컴퓨팅 성능을 절약하고 TTFT(첫 번째 토큰 수신 시간)를 크게 단축하고 있음을 의미

* 접두사 캐시 적중률
  + 적중률(캐시된 프롬프트 세그먼트를 활용하는 요청 비율)이 높을수록 컴퓨팅 성능을 절약하고 TTFT를 단축함
  + 적중률을 계산하는 프로메테우스 쿼리
    ```sql
    rate(vllm:prefix_cache_hits_total[1m]) / rate(vllm:prefix_cache_queries_total[1m])
    ```
<br>

### 2.4 요청 시퀀스 길이 분석

#### 2.4.1 토큰 갯 수에 따른 지연 시간

* 지연 시간 측정과 토큰 개수
  + 입력 시퀀스 길이(ISL)와 출력 시퀀스 길이(OSL)는 키-값 캐시의 메모리 사용량과 프리필 및 디코딩 단계에 필요한 연산 시간을 결정
  + 프리필과 디코딩 단계는 매우 비대칭적
  + 예
    - 최신 하드웨어에서는 1,000개 토큰으로 구성된 프롬프트(프리필)를 병렬로 처리하여 일반적으로 1초 이내에 완료
    - 그러나 1,000개 토큰(디코딩)을 생성하는 작업은 순차적으로 수행되므로 10초에서 100초 이상 소요

* ISL 및 TTFT
  + 프리필 시간은 입력 길이에 비례
  + 프롬프트가 매우 길면 엔진이 첫 번째 토큰을 생성하기 전에 처리해야 할 작업이 많아지므로 TTFT가 자연스럽게 높아짐
* OSL 및 ITL
  + 생성 시간은 생성되는 토큰 수에 비례
  + ITL이 40ms인 경우 1,000개의 토큰을 생성하는 데 40초가 걸리지만 100개의 토큰을 생성하는 데는 4초밖에 걸리지 않음

* 시퀀스 길이 확인하는 PromQL 쿼리
  ```sql
  # 95th percentile prompt length
  histogram_quantile(0.95, rate(vllm:request_prompt_tokens_bucket[5m]))
  # 95th percentile output length
  histogram_quantile(0.95, rate(vllm:request_generation_tokens_bucket[5m]))
  # Median output length
  histogram_quantile(0.5, rate(vllm:request_generation_tokens_bucket[5m]))
  ```

#### 2.4.2 검색 증강 생성(RAG) 관련 참고 사항

RAG 파이프라인의 일부인 vLLM 배포 환경
1. 짧은 사용자 쿼리가 벡터 데이터베이스에서 검색된 컨텍스트를 포함하는 방대한 프롬프트로 확장되는 경우가 많음
   + 이러한 대규모 프롬프트를 처리하는 데 필요한 컴퓨팅 자원으로 인해 TTFT가 높아질 수 있음
2. 벡터 데이터베이스 조회로 인해 요청에 추가적인 지연 시간이 발생할 수 있음
   + 이는 vLLM TTFT 지표에 반영되지 않음

RAG 워크로드에서 사전 채우기 비용을 줄이기
* 공유 시스템 프롬프트와 정적 명령어가 먼저 오도록 프롬프트를 구성
* vLLM의 접두사 캐싱이 이러한 공통 접두사를 요청 간에 캐시하고 재사용
* 이를 통해 중복 컴퓨팅 자원을 크게 줄임
<br>

### 2.5 분산 추론 전략 검토

#### 2.5.1 모델 크기와 GPU

* 모델 크기가 단일 GPU에서 처리하기에 너무 클 경우
  + 여러 GPU로 분산
  + 가장 일반적인 방법은 텐서 병렬 처리(TP)
    - TP는 각 레이어(예: 어텐션의 GEMM 또는 MLP) 내의 텐서 계산을 여러 GPU에 분산시켜 ITL을 개선
  + 모델을 여러 GPU로 분산시키면 각 GPU에 모델 가중치를 저장하는 데 필요한 메모리 양이 줄어둚
    - 이를 통해 키-값(KV) 캐시 공간을 확보하여 배치 크기를 늘리고 처리량을 높임

* 이러한 전략은 지속적인 동기화를 필요로 하므로 상당한 통신 오버헤드 발생
  + ITL이 예상치 못하게 높게 나오는 경우, `nvidia-smi topo -m` 명령어를 사용하여 하드웨어 구성을 확인
  + GPU가 NVLink와 같은 고속 인터커넥트 대신 PCIe를 통해 통신하는 경우, 이러한 동기화로 인한 성능 저하가 심각해짐

* 가장 효율적인 확장 방법
  + 모델의 여러 독립적인 복제본을 실행하는 것(데이터 병렬 처리) 또는 TP와 복제본을 결합한 하이브리드 방식
  + 이렇게 하면 토큰 생성 중 GPU 간 통신 병목 현상을 방지할 수 있으며, 일반적으로 시스템 전체 처리량을 높이는 데 더 나은 방법
  + 일반적으로 모델에 맞는 최소 TP 수준을 사용한 다음, 추가 용량을 위해 독립적인 복제본으로 확장
  + 복제본은 복제본 간 통신 오버헤드가 없으며 선형적으로 확장

#### 2.5.2 문제 해결 단계

vLLM 설정
* vLLM은 다양한 고급 설정을 제공하지만, 성능 문제는 CLI 매개변수 조정만으로는 해결되지 않는 경우가 많음
* 성능 향상을 위해서는 일반적으로 초기 진단 과정에서 파악된 하드웨어 또는 메모리 제약 조건을 해결 필요

병목 현상 파악 후, 효과적인 최적화 방안 고려
1. 모델 크기 최적화
   + 다른 설정을 조정하기 전에 현재 실행 중인 모델이 정말 필요한지 확인
   + 특정 작업에서는 8B 크기의 모델이 범용 70B 모델보다 성능이 뛰어나거나 비슷한 경우가 많으며, 지연 시간과 비용은 훨씬 적음
   + 워크로드가 명확하게 정의된 경우(분류, 데이터 추출, 요약 등), 작업에 특화된 작은 모델을 사용하면 성능 문제를 완전히 해결 가능

2. 하드웨어 확장
   + 시스템이 지속적으로 포화 상태인 경우(`num_requests_waiting` 값이 높은 경우), 복제본을 추가하여 부하를 분산
   + ITL이 병목 현상의 원인이라면 메모리 대역폭이 더 높은 GPU(예: L40S에서 H100으로)로 업그레이드하면 디코딩 단계 속도가 직접적으로 향상됨

3. 양자화 사용
   + FP16에서 FP8로 전환하면 모델의 VRAM 사용량이 절반으로 줄어들고, 각 순방향 패스에서 읽는 데이터 양이 감소하여 ITL이 향상
   + `--kv-cache-dtype fp8` 옵션을 사용하면 품질 손실을 최소화하면서 캐시 용량을 두 배로 늘릴 수 있음
   + 모델 제공업체가 Hugging Face에 사전 양자화된 체크포인트를 게시하므로 직접 양자화할 필요가 없을 수도 있음
   
4. 투기적 디코딩 구현
   + ITL이 주요 병목 현상이고 이미 양자화된 모델을 사용하는 고성능 하드웨어를 사용하고 있는 경우
     - 투기적 디코딩을 통해 순방향 패스당 여러 토큰을 생성
     - 출력 품질 저하 없이 디코딩 처리량을 효과적으로 높임
   + vLLM은 네이티브 MTP, EAGLE3, 드래프트 모델, n-gram 투기 디코딩을 포함한 여러 가지 방법을 지원
     - 레드햇은 인기 있는 오픈 소스 모델에 대해 [Hugging Face에서 사전 학습된 여러 투기자](https://huggingface.co/collections/RedHatAI/speculator-models)를 제공
     - [`Speculators` 라이브러리](https://github.com/vllm-project/speculators)를 사용하여 자체 정렬된 드래프터도 학습 가능

5. 분산 개선
   + 멀티 GPU 배포에서 상호 연결 오버헤드가 발생하는 경우 텐서 병렬(TP) 수준을 낮추는 것을 고려
   + 일반적으로 TP=2로 설정된 두 개의 독립적인 복제본이 TP=4로 설정된 하나의 인스턴스보다 처리량과 스케줄링 유연성 측면에서 더 나은 성능을 보임
   + 하드웨어에 NVLink가 전혀 없는 경우 GPU당 TP=1을 유지하고 순수 데이터 병렬 또는 파이프라인 병렬 처리를 실행하는 것이 최선의 선택일 수 있음

> [!NOTE]
> 하드웨어 한계 및 워크로드 형태를 고려하여 메트릭을 분석하면 시행착오를 줄이고 수요 증가에 따라 LLM 인프라의 성능을 유지할 수 있습니다.

## 3. 요약 및 다음 단계

단일 vLLM 인스턴스의 문제 해결 방법은 위와 갈으며 이제는 더 큰 그림을 살펴볼 준비가 되었습니다. 성능 튜닝은 단일 GPU에서 멈추지 않습니다. 실제 운영환경에서는 다음 단계로 스케일 아웃이 고려되는 경우가 많습니다. 

쿠버네티스와 레드햇 오픈시프트 AI와 같은 스케일 아웃 환경에 최적화된 llm-d를 사용한 분산 추론 진단 및 문제 해결 전략이 필요로 합니다. 노드 간 상호 연결 오버헤드를 해결하든, 최적의 로드 밸런싱을 위해 다중 복제 큐 깊이를 최적화하든, 확장 가능한 고성능 LLM 인프라를 유지하기 위해 llm-d를 통해 최적화하는 방법이 필요로 합니다.
