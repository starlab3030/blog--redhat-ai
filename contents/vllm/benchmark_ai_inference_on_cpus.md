## 1. CPU 기반 AI 추론 서비스

### 1.1 벤치마킹 프레임워크의 필요성

기업들이 대규모 언어 모델(LLM) 배포의 총 소유 비용(TCO)을 최적화하기 위해, 기업은 기존의 CPU 인프라와 특정 추론 워크로드에 필요한 GPU 리소스를 함께 활용하는 것이 전략적 과제가 되었습니다. 그러나 이러한 전략을 검증하려는 인프라 팀은 혼란스러운 벤치마킹 환경에 직면하고 있습니다. 현재 성능 평가에는 업계 표준이 부재하며, 하드웨어 공급업체들은 투명하고 재현 가능한 방법론 없이 최상의 처리량 지표만을 발표하는 경우가 많습니다.

실제 운영 환경에 적용하기 위한 아키텍처으로 신뢰를 구축하려면, CPU 추론 성능을 평가하는 방식을 표준화해야 합니다. 엔지니어링 팀은 이상화된 마케팅 데이터에 의존하는 것을 넘어, 최적화되지 않은 트래픽, 가변적인 토큰 분포, 실제 하드웨어 포화 한계 등을 정확하게 반영하는 엄격하고 반복 가능한 벤치마킹 프레임워크가 필요합니다.

이러한 표준을 수립하기 위해 레드햇의 OCTO 신기술 및 성능/확장성 엔지니어링 팀은 CPU 기반 vLLM 성능 평가 프레임워크를 개발했습니다. 이 자동화된 오픈 소스 테스트 도구 모음은 앤서블과 GuideLLM을 사용하여 성능 벤치마킹을 표준화하고, 기업이 CPU 용량을 자신 있게 평가하고, 하드웨어 병목 현상을 파악하고, 프로덕션 최적화를 검증할 수 있도록 보편적으로 검증 가능한 기준선을 설정합니다.

> [!NOTE]
> 이 글에는 레드햇에 오픈 소스 커뮤니티에서 활발히 개발 중인 기술에 대한 게시물이 포함되어 있습니다. 레드햇은 개발 중인 내용을 조기에 자주 공유하는 것을 중요하게 생각하지만, 별도로 명시되지 않는 한 여기에 공유된 기술 및 사용 방법은 지원되는 제품에 포함되지 않으며 향후에도 지원될 예정이 없음을 알려드립니다.
<br>

### 1.2 핵심 요약

* 표준의 필요성
  +  현재 특정 추론 워크로드를 CPU에서 실행하는 타당성을 평가하는 데에는 파편화되고 불투명한 벤치마크가 사용되어 실제 운영 환경을 제대로 반영하지 못하고 있음
* 해결책
  + 재현 가능한 다중 테넌트 엔터프라이즈 환경을 시뮬레이션하기 위해 앤서블과 GuideLLM을 사용하여 구축된 표준화된 오픈 소스 벤치마킹 프레임워크 (vllm-cpu-perf-eval) 사용
* 영향
  + 인프라 엔지니어에게 기본 용량, 트래픽 변동성 및 최적화 효과를 측정하여 CPU 도입 위험을 정확하게 줄일 수 있는 통일된 테스트 표준을 제공
* 프레임워크
  + 기준 용량, 현실적인 변동성 분석, 생산 최적화 검증을 포괄하는 엄격한 3단계 테스트 방법론
<br>
<br>

## 2. vLLM을 위한 CPU 성능 테스트 표준화

이 섹션에서는 소규모 언어 모델(SLM)을 실행할 때 vLLM 프레임워크(CPU 모드)의 성능을 평가하기 위한 단계, 지표 및 절차를 설명합니다.

### 2.1 표준 벤치마킹 유틸리티

#### 2.1.1 성능 평가 도구: GuideLLM

* 테스트 시 버전: v0.6.0
* GuideLLM의 역할
  + 벤치마크 유틸리티
    - 성능 측정을 위한 프레임워크와 지표를 제공
  + 부하 발생기
    - 실제 상황 또는 스트레스 상황에서 시스템을 테스트하는 데 필요한 사용자 요청 및 작업 부하를 시뮬레이션

> [!NOTE]
> GuideLLM은 대규모 언어 모델(LLM)을 평가하기 위한 플랫폼/워크로드 생성기 역할을 수행하며, 실제 추론 워크로드를 시뮬레이션하여 사용자가 다양한 하드웨어 구성에 LLM을 배포할 때의 성능, 리소스 요구 사항 및 비용 영향을 평가할 수 있도록 지원합니다.
<br>

### 2.2 평가된 모델

고속 채팅부터 복잡한 전문가 혼합(MoE) 확장까지 모든 것을 스트레스 테스트하기 위해 다양한 디코더 전용 아키텍처를 선정

|AI 모델 종류     |대표 모델                 |주요 응용 분야 |설명|
|:---           |:---                   |:---        |:---|
|Llama 3        |Llama-3.2-1B-Instruct  |Prefill이 높음|최신 Llama 아키텍처; 고밀도 입력 처리 벤치마크|
|Llama 2        |TinyLlama-1.1B-Chat    |소규모 효율성   |엣지/저코어 환경에 최적화된 리소스 효율적인 버전|
|IBM Granite    |granite-3.2-2b-instruct|기업을 위한 기준선|레드햇의 대표적인 균형 잡힌 프로덕션 지침 준수 모델|
|Qwen 3         |Qwen3-0.6B             |고효율        |메모리 사용량을 최소화하면서 처리량을 극대화하도록 최적화|
|Transformer MoE|gpt-oss-20b            |확장성 테스트   |128k 컨텍스트 창에서 기준 CPU 용량 및 확장 동작을 설정하기 위한 21B(활성 3.6B) 파라미터|
<br>

### 2.3 표준화된 워크로드 프로파일

#### 2.3.1 워크로드 프로파일 설명

* 다양한 유형의 고강도 컴퓨팅 부하 조건에서 모델을 균일하게 벤치마킹하기 위해 5개의 서로 다른 프로파일 정의
* 프로파일들은 입력 시퀀스 길이(ISL)와 출력 시퀀스 길이(OSL)의 균형에 따라 엄격하게 구분
  + 이를 통해 사전 입력 작업이 많은 시나리오(긴 입력 컨텍스트 처리)와 디코딩 작업이 많은 시나리오(긴 출력 텍스트 생성) 모두를 벤치마킹
  
#### 2.3.2 워크로드 프로파일 목록

|워크로드 유형|ISL:OSL|설명|
|:---|:---:|:---|
|채팅    |512:512  |균형 잡힌 사전 입력/디코딩, 일반적인 대화형 AI|
|요약    |2048:256 |중간 수준의 맥락, 요약 작업|
|코드 생성|1024:1024|균형 잡힌 코드 생성 시나리오|
|조각    |8192:512 |긴 컨텍스트 사전 채우기, 검색 증강 생성|
|추리    |256:2048 |긴 출력 디코딩, 사고 연쇄 추론|
<br>

### 2.4 전체 스택 측정: 클라이언트, 서버 및 시스템 지표

실제 성능 프로파일을 파악하기 위해 세 가지 원격 측정 데이터를 동시에 추적합니다. 이를 통해 예를 들어 서버의 요청 큐 깊이 급증이 최종 사용자의 첫 번째 토큰 수신 시간 저하와 어떻게 연관되는지 확인할 수 있습니다.

#### 2.4.1 클라이언트 측 지표 (사용자 경험)

* Time-to-First-Token (TTFT): 인지된 응답성
* Inter-Token Latency (ITL): 모델의 "읽기 속도"의 일관성
* End-to-End Latency (E2E): 전체 응답에 소요되는 왕복 시간
* 총 토큰 처리량: 사용자에게 전달된 데이터의 총량

#### 2.4.2 서버 측 지표 (엔진 성능)

* TTFT, ITL, E2E 지연 시간 및 총 토큰 처리량 (서버 관점)
* KV 캐시 활용률: 컨텍스트에 메모리가 얼마나 효율적으로 사용되는지를 추적
* 요청(req) 큐 깊이: 엔진이 포화 상태에 이르는 병목 현상을 식별
* 토큰 생성 속도: 네트워크 오버헤드와 무관한 CPU 환경의 기본 생성 용량
* 평균 프롬프트/생성 길이: 처리량 변동에 대한 맥락을 제공

#### 2.4.3 시스템 수준 지표 (인프라 상태)

* CPU 사용률(%): 코어 포화도를 모니터링하고 컴퓨팅 또는 메모리 대역폭 한계에 도달했는지 여부를 확인
* 메모리 사용량(GB): 모델 가중치와 동적 KV 캐시 버퍼를 포함한 전체 사용량
<br>

## 3. 레드햇의 3단계 테스트 방법론

### 3.1 성능 평가 프레임워크

#### 3.1.1 vLLM CPU 성능 평가 프레임워크

* 기준 성능 측정, 실제 변동성 분석, 운영 환경 최적화 평가를 분리하는 구조화된 3단계 테스트 접근 방식을 사용
* 이 방법론은 다양한 테스트 스위트에 적용할 수 있도록 설계되었지만, 현재는 동시 부하 테스트에만 완전히 구현되어 있음

#### 3.1.2 주요 이점

* 재현 가능한 기준선: 고정된 작업량은 동일한 조건에서의 비교를 위한 변동성을 제거
* 현실적인 인사이트: 다양한 작업 부하를 통해 실제 트래픽 패턴을 시뮬레이션
* 운영 환경 검증: 캐싱 비교를 통해 최적화 효과를 정량화
* 명확한 진행 과정: 각 단계는 이전 단계의 결과를 바탕으로 진행
<br>

### 3.2 3단계 테스트 접근 방식

#### 3.2.1 1단계: 기초 검사

* 목표: 최대 처리량 및 하드웨어 포화 지점 파악
* 방법: 합성 데이터셋을 사용하고, 입력 시퀀스 길이(ISL)와 출력 시퀀스 길이(OSL)를 고정하며, 접두사 캐싱(vLLM)을 비활성화
* 임계 작업 부하 비율 설정
  |워크로드 유형|ISL:OSL|설명|
  |:---|:---:|:---|
  |채팅    |512:512  |균화형 AI|
  |요약    |8192:512 |긴 컨텍스트 검색|
  |코드 생성|1024:1024|코드 생성|
  |조각    |2048:256 |문서 처리|
  |추리    |256:2048 |밀집 임베딩|

#### 3.2.2 2단계: 현실적인 테스트

* 목표: 인위적인 일관성을 깨뜨리고 분산에 민감한 모델을 측정한다.
* 방법: 실제 토큰 분포의 변동성을 도입하여 P95/P99 배치 완료 편차를 측정합니다.
* 임계 작업 부하 비율 설정 예시
  |워크로드 유형|ISL:OSL|
  |:---   |:---|
  |채팅    |입력 512±128, 출력 256±64|
  |요약    |입력 1024±512, 출력 256±128|
  |코드 생성|입력 512±128, 출력 4096±1024|
  
#### 3.2.3 3단계: 생산 테스트

* 목표: 생산 단계 최적화의 영향을 정량화
* 방법: 실제 데이터 세트로 전환하고 접두사 캐싱(vLLM)을 활성화하여 최종 지연 시간 특성을 확인
<br>

### 3.3 벤치마크 실행 구성

* 각 실행은 600초 범위 내에서 진행
* 실행 별 구간 구성
  + 30초의 워밍업
  + 엄격한 경계 설정이 적용되는 540초의 측정 구간
  + 그리고 30초의 쿨다운
<br>
<br>

## 4. 테스트 환경 및 벤치마크 결과 평가

### 4.1 표준호된 테스트를 위한 격리된 아키텍처

#### 4.1.1 격리된 아키텍처가 필요한 이유

* 성능 데이터의 무결성을 보장하기 위해 레드햇은 "관심사 분리(Strict Isolation of Concerns)" 원칙을 엄격히 준수
* 부하 생성 프로세스가 추론 엔진과 동일한 컴퓨팅 또는 메모리 리소스를 공유하지 않도록 설정
  + 공유 오버헤드는 측정하고자 하는 CPU 집약적 LLM 지표를 왜곡할 수 있음
* 격리 환경 지원을 위해 프레임워크는 두 가지 배포 모드를 제공

#### 4.1.2 모드 1: 물리적 노드 분리 (이상적)

벤치마킹의 최고 기준으로, 고속 네트워크 세그먼트로 연결된 듀얼 노드 구성을 활용

* 부하 발생기 노드
  + GuideLLM 벤치마킹 제품군을 실행하는 전용 시스템
* 테스트 대상 장치(DUT, Device-Under-Test)
  + Intel Xeon 또는 AMD EPYC 플랫폼에서 vLLM을 실행하는 격리된 시스템
  + 이를 통해 하드웨어 수준에서 "노이즈가 발생하는 이웃" 효과의 가능성을 제거

#### 4.1.3 모드 2: 노드 내 소켓 격리 (논리적 분리)

* 물리적 노드를 별도로 사용할 수 없는 경우 소켓 수준의 격리를 활용
  + 이 모드에서는 최신 엔터프라이즈 CPU의 멀티 소켓 아키텍처(예: AWS c8i.metal-48xl 또는 m8a.metal-48xl 인스턴스)를 활용
* 소켓별 구성
  + 소켓 0: GuideLLM 및 시스템 오버헤드 전용
  + 소켓 1(DUT): vLLM 컨테이너 전용 소켓

> [!NOTE]
> 앤서블은 코어 고정 및 NUMA 인식 메모리 할당을 보장하여 부하 생성기와 추론 엔진이 독립적인 L3 캐시 및 메모리 컨트롤러에서 작동하도록 합니다.
<br>

### 4.2 메모리 효율성

#### 4.2.1 KV 캐시 크기 표준화

* (모델 가중치 다음으로) 키-값(KV) 캐시는 추론 파이프라인에서 가장 많은 메모리를 소비하는 요소
  + 캐시 크기가 부족하면 시스템은 메모리 부족(OOM) 오류로 인해 충돌
  + 캐시가 너무 크면 값비싼 기업 자원을 낭비
* "엔터프라이즈 AI 현실"을 구현하기 위해, 추측에 의존하는 방식에서 벗어나 적정 크기의 캐시 전략 필요
  + 재현성과 효율성을 제공할수 있도록, 필요한 정확한 크기를 계산하기 위해, 투명한 수학적 접근 방식을 사용
    ```
    총 요소 수 = 2(KV 모두 저장을 의미) * 레이어 수 * 총 토큰 수 * KV 헤드 수 * 헤드 크기
    총 바이트 수 = 총 요소 수 * dtype_size (참조: bfloat16의 경우 dtype_size는 2-바이트)
    ```

#### 4.2.2 레드햇의 사이즈 전략

과도한 리소스 할당 없이 트래픽 급증 시에도 안정성을 보장하는 방법론
1. 기본 요구 사항
   + 특정 모델 아키텍처를 기반으로 요청당 크기를 계산
2. 선형 확장
   + 계획된 동시 처리량으로 곱함
   + 레드햇의 표준 테스트 스위트는 최대 32개의 동시 요청을 사용
3. 안전을 위한 여유분 설정
   + 1.25배(25%)의 버퍼를 적용
   + 이 보호 계층은 토큰 분포의 변동으로 인해 즉각적인 오류가 발생하는 것을 방지

#### 4.2.3 KV 캐시를 위한 메모리 설정 예시

**모델**
* Llama-3.2-1B Chat 모델을 사용
  + 레이어 수: 16
  + 숨겨진 차원: 2048
  + 어텐션 헤드 수: 32
  + KV 헤드 수: 8

**토큰**
* 1024개의 토큰 워크로드 설정
  + ISL:512
  + OSL:512

**요청당 크기 계산식**
* 총 요소 수
  ```
  16,777,216(총 요소 수) = 2 * 16(레이어 수) * 1024(총 토큰 수) * 8(KV 헤드 수) * 2048(숨겨진 차원)/32(어텐션 헤드 수)
  ```
* 총 바이트 수 (FP16의 경우 2-바이트)
  ```
  0.03125GB = 16,777,216 * 2 / (1024 * 1024 * 1024)
  ```

**필요한 메모리 크기**
* 기본 용량: 요청당 0.03125GB
* 규모: 동시 요청 32개
* 보호: 안전 여유분 1.25배
* 결과: 1.25GB 필요
  ```
  1.25GB = 0.03125 * 32 * 1.25 
  ```
* 최적의 시스템 정렬을 위해 2GiB로 구성

> [!NOTE]
> 이러한 적정 규모 접근 방식을 통해 기업은 CPU 플랫폼에서 밀도를 극대화하여 소켓당 더 많은 모델을 실행하는 동시에 100% 신뢰성 목표를 유지할 수 있습니다.
<br>

### 4.3 벤치마크 결과의 안정성 및 재현성 평가

#### 4.3.1 벤치마크 실행의 안정성 평가

* 벤치마크를 두 번 실행했을 때 완전히 다른 결과가 나온다면 그 벤치마크는 무용지물
* 레드햇이 제공하는 프레임워크는, 과학적으로 엄격한 재현성을 보장하도록 하기 위해 평균값만 보고하는 것이 아니라, 변동계수(CV)를 사용하여 모든 벤치마크 실행의 안정성을 평가
* 변동 계수(CV) 계산식
  ```
  CV = (𝝈 / µ) * 100
  ```

#### 4.3.2 신뢰도 등급 척도

|CV 범위     |반복성 등급|
|:---:      |:---|
|< 1.0%     |우수 (A+)|
|1.0% – 3.0%|좋음 (A~B+)|
|3.0% – 5.0%|허용 가능 (B)|
|> 5.0%     |불량 (C)|
* 이 척도를 통해 엔지니어는 하드웨어 구성 또는 최적화가 "생산 준비 완료" 상태인지 아니면 환경적 잡음의 영향을 받고 있는지 신속하게 판단 가능
